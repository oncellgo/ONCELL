# service-plan — 참여 활성화 기획 프로토콜

*"성도가 오늘 앱을 한 번 더 열게 만드는 것은 무엇인가?"* — 단, 숫자가 아닌 **영적 성장·공동체 유대**를 해치지 않는 선에서.

## 1) 현재 참여 지점 점검
- 큐티(새벽영상+매일성경+묵상노트) · 장소예약(30분 슬롯) · 공지/일정(events.json 통합) · 대시보드 위젯.
- 각 지점의 **진입 마찰**을 열거 (로그인 요구/필수정보/권한/404/로딩 지연).

## 2) 사용자 3유형
- **일반 성도** (10–70대, 모바일, 짧은 접속)
- **목회자/리더** (관리자, 공지·모임 기획)
- **Seeker** (새가족, 부담 없이 둘러보기)
세 유형 각각에서 "한 번 더 열게 만드는 트리거"를 분리해서 본다.

## 3) 분석 프레임 (필수)
1. **Hook 모델** — Trigger(외부/내부) → Action(최소 노력) → Variable reward → Investment.
2. **Job-to-be-done** — "이 기능으로 내가 어떤 작업을 해치우려 하는가?"
3. **Friction audit** — 클릭 수, 입력 수, 대기 시간.

## 4) 추천 기법 (신중히 적용)
- **가시적 연속성**: 큐티 스트릭·월간 출석 히트맵
- **부드러운 알림**: 푸시·이메일 — 단 주 2–3회 넘지 않게
- **소셜 증명**: 최근 묵상노트 익명 발췌·공동체 참여 수
- **작은 승리**: 완료 뱃지·애니메이션(과도 금지)
**주의**: 게이미피케이션은 신앙 자체를 경쟁으로 만들면 실패. 공동체 격려 표현으로만.

## 5) 산출물 형식
- **진단 요약** (3–5문장, 현재 상태를 가감없이)
- **가설 1–3개** — "X가 참여율을 올릴 것이다. 이유는…"
- **가장 작은 실험** — 개발 1–2일 내 착수 가능한 MVP
- **리스크** — 역효과 시나리오
- **측정 지표** — DAU, 리텐션보다 **의미있는** 지표 (예: 주 3회 이상 큐티 접속자 수)

## 6) 바꾸지 말 것
- 인증/승인 흐름(`signupApproval`, `signupRequiredFields`)은 목회적 결정 — 설정으로만 조절.
- 관리자/일반 사용자 권한 경계.
- plan.md의 통합 DB 원칙(events.json type).

## 7) 핵심 도메인 규칙 (건드리지 말 것)

### 큐티 완료는 **오늘(SG 기준)만** 가능
- 과거 날짜: **소급 완료 금지** — 묵상노트 작성/"✓ 큐티 완료" 토글 불가. 조회(읽기)만 허용해서 다른 요일 말씀을 훑어볼 수는 있으나, 완료 처리·저장은 오늘 분에만 귀속.
- 미래 날짜: **선지급 금지** — 선택조차 못 하도록 UI 비활성. 영상/본문도 미래 것은 표시하지 않는다.
- 기준 날짜: 반드시 **`getSGTodayKey()`** (SG wall-clock) — 브라우저 로컬 TZ 기반의 `new Date().getDate()` 금지.
- 대시보드 요일 pill 은 과거·오늘 모두 클릭으로 이동 가능하지만, **과거 pill 은 "읽기 모드" 로만 진입**하도록 구현(완료 CTA 숨김/비활성).
- **왜**: 큐티는 "오늘 하나님과의 만남"이라는 영적 행위. 사후 완료 표시는 기록의 의미를 왜곡시키고, 스트릭·참여 집계를 신뢰 불가능하게 만듦. 목회 관점에서도 "오늘의 은혜"를 강조해야 유도가 유의미.

### 성경통독은 catch-up 허용
- 과거/미래 모두 완료 토글 가능. 통독은 "계획된 분량을 꾸준히" 개념이라 하루 놓치면 소급 가능.
- 이 차이는 **의도**. 큐티와 같은 취급으로 맞추지 말 것.

### 구역모임교안 "N째 주" 매핑 기준 — **금요일 포함 주**
- 교회 관행: 게시글 제목 `YYYY년 N월 M째 주_금요 구역예배지` 의 **M째 주**는 **해당 월의 M번째 금요일이 포함된 주**(일~토)를 가리킨다. "해당 월의 M번째 일요일" 이 아님.
- UI 매핑: 그 주의 **다음 일요일 pill** 에 콘텐츠 노출. pill 은 직전 일~토 주의 자료를 보여주는 구조이므로 "N번째 금요일이 있는 주 + 2일(일요일)" 이 pill 의 dateKey.
- 예) 2026-04: 1st Fri = 4/3 → pill **4/5** 에 "4월 첫째 주" 노출. 5th Fri 는 5/1 (월 벗어남) → "5째 주" 부재.
- 주의 — 월초가 토/일인 달은 두 룰(일요일 기준 vs 금요일 포함 주)이 **1주 차이**. 예) 2026-02-01(일): 첫 금요일 = 2/6 → pill **2/8** 에 "2월 첫째 주". 일요일 기준(2/1)과 다름.
- 파서 위치: `pages/api/cell-worship.ts` 의 `sundayAfterNthFriday(year, month, nth)`. 기존 `nthSundayOf` 는 이 룰과 맞지 않아 교체됨.
- KV 캐시 키 변경 시 version 을 bump 해 기존 결과를 invalidation 할 것 (`cell_worship_list_cache_vN`).

### 구현 체크리스트 (코드 수정 시)
- `/qt` 페이지: `selectedDateKey !== getSGTodayKey()` 이면 "묵상노트 열기" / "✓ 큐티 완료" 토글 → **읽기 전용 배지로 치환**(예: "📖 과거 큐티 보기").
- `/api/qt-notes` POST: 날짜가 오늘이 아니면 **400 에러** 반환. 클라이언트 UI 막는 것만으로는 부족, 서버에서도 방어.
- 대시보드 pill: 과거 pill 의 title 툴팁에 "보기만 가능" 명시.
- 이 규칙을 바꾸려면 목회자 승인 필요 — 코드만 고치고 넘기지 말 것.

## 8) 소속/연계
- **소속 에이전트**: `service-planner` (`.claude/agents/service-planner.md`) — 이 스킬은 그 에이전트의 실행 프로토콜.
- **화면·레이아웃 수정** → `ux-designer` 에이전트 / `ux-review` 스킬에 위임.
- **기술 구현·비용 추정** → `tech-optimizer` 에이전트 / `tech-audit` 스킬에 위임. 이 스킬은 **스펙만 생성**, 코드 작성 금지.
