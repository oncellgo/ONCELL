# service-plan — 참여 활성화 기획 프로토콜

*"성도가 오늘 앱을 한 번 더 열게 만드는 것은 무엇인가?"* — 단, 숫자가 아닌 **영적 성장·공동체 유대**를 해치지 않는 선에서.

## 1) 현재 참여 지점 점검
- 큐티(새벽영상+매일성경+묵상노트) · 장소예약(30분 슬롯) · 공지/일정(events.json 통합) · 대시보드 위젯.
- 각 지점의 **진입 마찰**을 열거 (로그인 요구/필수정보/권한/404/로딩 지연).

## 2) 사용자 3유형
- **일반 성도** (10–70대, 모바일, 짧은 접속)
- **목회자/리더** (관리자, 공지·모임 기획)
- **Seeker** (새가족, 부담 없이 둘러보기)
세 유형 각각에서 "한 번 더 열게 만드는 트리거"를 분리해서 본다.

## 3) 분석 프레임 (필수)
1. **Hook 모델** — Trigger(외부/내부) → Action(최소 노력) → Variable reward → Investment.
2. **Job-to-be-done** — "이 기능으로 내가 어떤 작업을 해치우려 하는가?"
3. **Friction audit** — 클릭 수, 입력 수, 대기 시간.

## 4) 추천 기법 (신중히 적용)
- **가시적 연속성**: 큐티 스트릭·월간 출석 히트맵
- **부드러운 알림**: 푸시·이메일 — 단 주 2–3회 넘지 않게
- **소셜 증명**: 최근 묵상노트 익명 발췌·공동체 참여 수
- **작은 승리**: 완료 뱃지·애니메이션(과도 금지)
**주의**: 게이미피케이션은 신앙 자체를 경쟁으로 만들면 실패. 공동체 격려 표현으로만.

## 5) 산출물 형식
- **진단 요약** (3–5문장, 현재 상태를 가감없이)
- **가설 1–3개** — "X가 참여율을 올릴 것이다. 이유는…"
- **가장 작은 실험** — 개발 1–2일 내 착수 가능한 MVP
- **리스크** — 역효과 시나리오
- **측정 지표** — DAU, 리텐션보다 **의미있는** 지표 (예: 주 3회 이상 큐티 접속자 수)

## 6) 바꾸지 말 것
- 인증/승인 흐름(`signupApproval`, `signupRequiredFields`)은 목회적 결정 — 설정으로만 조절.
- 관리자/일반 사용자 권한 경계.
- plan.md의 통합 DB 원칙(events.json type).

## 7) 소속/연계
- **소속 에이전트**: `service-planner` (`.claude/agents/service-planner.md`) — 이 스킬은 그 에이전트의 실행 프로토콜.
- **화면·레이아웃 수정** → `ux-designer` 에이전트 / `ux-review` 스킬에 위임.
- **기술 구현·비용 추정** → `tech-optimizer` 에이전트 / `tech-audit` 스킬에 위임. 이 스킬은 **스펙만 생성**, 코드 작성 금지.
